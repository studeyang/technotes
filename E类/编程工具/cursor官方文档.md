# 一、Agent

## 1.1 概览

**智能体 的工作原理**

一个智能体由三个组件构成：

1. Instructions：引导智能体行为的 system prompt 和 [rules](https://cursor.com/docs/rules)
2. Tools：文件编辑、代码库搜索、终端执行等
3. Model：你为该任务选择的智能体模型

**即时发送消息**

当你使用 Ctrl+Enter 即时发送时，你的消息会附加到当前对话中最近的一条用户消息后面，并立即处理，而无需在队列中等待。

- 你的消息会附加在工具执行结果后并立即发送
- 有助于在紧急跟进时获得更快速的响应
- 当你需要中断或重定向 Agent 当前任务时使用此方式

## 1.2 规划模式

**工作原理**

1. Agent 会先提出澄清性问题，以更好地理解你的需求
2. 在你的代码库中进行检索，收集相关上下文
3. 制定一个完整的实现方案
4. 你可以通过对话或 Markdown 文件来审阅并编辑方案
5. 准备好后，点击开始构建方案

方案默认保存在你的主目录中。点击 “Save to workspace” 可将其移动到你的工作区，以便后续查阅、团队共享和文档记录。

**何时使用 Plan 模式**

Plan 模式最适合：

- 具有多种可行方案的复杂功能
- 涉及大量文件或系统的任务
- 需求不清晰、需要先探索再弄清工作范围的任务
- 你希望先审查整体方案的架构决策

对于一些简单修改或你已经做过很多次的任务，可以直接使用 Agent 模式。

**从计划重新开始**

有时 Agent 生成的结果与你的预期不符。与其用后续提示一点点修修补补，不如回到最初的计划。

撤销这些更改，把计划写得更具体、更清晰，然后再重新运行。这样通常比修复一个进行中的 Agent 更快，输出也更干净利落。

## 1.3 提示工程

**为 Agent 编写提示**

- @ 提及：在聊天输入框中键入 `@`，即可为你的提示附加特定上下文。在 `@` 之后开始输入，Cursor 会显示匹配的建议项。
- 图像输入：在你的提示中附加图像，为 UI 工作、调试和设计实现提供视觉上下文。

## 1.4 调试模式

**何时使用 Debug Mode**

Debug Mode 最适合用于：

- 你能复现但搞不清原因的 bug：当你知道有问题，但光看代码看不出原因时
- 竞争条件和时序问题：依赖执行顺序或异步行为的问题
- 性能问题和内存泄漏：需要运行时分析才能理解的问题
- 曾经能用但现在失效的回归问题：当你需要追踪到底改了什么时

当标准 Agent 交互难以解决某个 bug 时，Debug Mode 会基于运行时证据，而不是凭猜测来给出修复方案，提供一种不同的排查方式。

**工作原理**

1. 探索并提出假设：Agent 会探索相关文件、构建上下文，并针对潜在根本原因生成多个假设。
2. 添加埋点：Agent 会添加日志语句，将数据发送到运行在 Cursor 扩展中的本地调试服务器。
3. 复现 bug：Debug Mode 会让你复现 bug，并提供具体步骤。这能让你始终参与调试流程，并确保 Agent 捕获到真实的运行时行为。
4. 分析日志：复现完成后，Agent 会审查收集到的日志，根据运行时证据识别实际根本原因。
5. 实施针对性修复：Agent 会进行有针对性的修复，直接解决根本原因，通常只需改动几行代码。
6. 验证并清理：你可以重新运行复现步骤来验证修复。一旦确认无误，Agent 会移除所有埋点。

**调试模式使用建议**

- 提供详细上下文：你对 bug 及其复现方式描述得越详细，Agent 的调试信息收集效果就会越好。请包含错误信息、堆栈跟踪以及具体操作步骤。
- 严格按照复现步骤执行：执行 Agent 提供的步骤，以确保日志能捕获到真实问题。
- 必要时多次复现：多次复现 bug 可以帮助 Agent 识别诸如竞态条件之类的棘手问题。
- 明确预期行为与实际行为：帮助 Agent 理解「应该发生什么」与「实际发生了什么」。

## 1.5 工具搜索

**索引的工作原理**

Cursor 会将你的代码拆分为有意义的片段 (函数、类、逻辑块) ，将每个片段转换为捕捉其语义的向量嵌入，并将结果存储在向量数据库中。你进行搜索时，查询会使用相同的模型转换为向量，并与已存储的嵌入进行匹配。

当你打开一个工作区时，会自动开始建立索引。索引完成 80% 后即可使用语义搜索。索引通过每 5 分钟自动同步一次保持最新状态，并且只处理已更改的文件。

| 变更类型   | 操作                     |
| :--------- | :----------------------- |
| 新文件     | 自动加入索引             |
| 修改的文件 | 旧嵌入会被删除并重新生成 |
| 已删除文件 | 从索引中移除             |

**配置**

在 Cursor Settings > Indexing 中检查索引状态或触发重新索引。

Cursor 会为除 [忽略文件](https://cursor.com/docs/reference/ignore-file) (`.gitignore`、`.cursorignore`) 之外的所有文件建立索引。忽略大型生成文件或内容文件有助于提升搜索准确性。

要查看已建立索引的文件路径：Cursor Settings > Indexing & Docs > View included files。

**Agent 如何组合搜索工具**

Agent 会根据你的提示选择合适的工具：

| 提示风格         | 使用的工具                     | 示例                                     |
| :--------------- | :----------------------------- | :--------------------------------------- |
| 指定符号或字符串 | Grep                           | 「查找所有导入 `PaymentService` 的文件」 |
| 概念或行为       | 先语义搜索，再用 Grep 补充细节 | 「我们的应用如何处理支付失败？」         |
| 复杂探索         | 多次搜索、读取文件、跟踪引用   | 「梳理从结账到确认邮件的整个数据流」     |

你不需要自己选工具。只要描述你的需求，Agent 会自行决策。对于复杂任务，它会串联多种搜索操作：先用语义搜索找到入口点，再用 Grep 追踪引用，最后通过读取文件构建完整上下文。

**Explore 子代理**

Agent 可以创建一个 [Explore 子代理](https://cursor.com/docs/subagents)，它在自己的上下文窗口中运行，并使用更快的模型。它会并行执行大量搜索，而不会让主对话变得臃肿，只返回相关结果。

当 Agent 判断某个任务需要进行广泛搜索时，会自动使用 Explore 子代理。你也可以直接请求它：“使用一个子代理查找所有我们校验用户输入的地方。”

这有助于上下文管理。在许多文件中搜索会生成大量上下文。子代理通过总结结果，而不是直接倾倒原始文件内容，来让主对话保持聚焦。

## 1.6 并行代理

**基本 Worktree 用法**

在 Cursor 中使用 worktree 的最基本方式是在一个 worktree 中运行单个 agent。

![Basic worktree usage](https://technotes.oss-cn-shenzhen.aliyuncs.com/2026/202603271355058.png)

当 agent 运行完成后，点击“Apply”按钮，将 agent 的更改应用到本地分支。

**Best-of-N —— 同时使用多个模型**

worktrees 的一项强大功能是可以将同一条提示同时运行在多个模型上。

![Best-of-N 基本用法](https://technotes.oss-cn-shenzhen.aliyuncs.com/2026/202603271359944.png)

提交提示后，你会看到两张卡片，每个模型对应一张。点击不同卡片以查看各个 Agent 所做的更改。然后点击“Apply”按钮，将更改应用到已检出的分支。

在多个模型上运行相同的提示词在以下场景尤其有用：

- 复杂问题，不同模型可能会采用不同的解决思路
- 比较代码质量，在不同模型家族之间寻找最优解
- 发现边界情况，一个模型可能漏掉而另一个模型能捕获到
- 基准测试，找出哪些模型最适合你的特定代码库

让多个模型同时尝试解决同一个问题，然后选取最佳结果，可以显著提升最终输出质量，尤其是在处理更难的任务时。

# 二、自定义

## 2.1 Plugin

一个插件可以捆绑以下组件的任意组合：

| 组件           | 描述                                     |
| :------------- | :--------------------------------------- |
| **规则**       | 持久性的 AI 指导和编码标准 (`.mdc` 文件) |
| **技能**       | 用于复杂任务的专门化智能体能力           |
| **Agents**     | 自定义智能体配置和提示词                 |
| **命令**       | 可由智能体执行的命令文件                 |
| **MCP 服务器** | Model Context Protocol 集成              |
| **钩子**       | 由事件触发的自动化脚本                   |

**创建插件**

你可以从 [插件模板代码仓库](https://github.com/cursor/plugin-template) 开始，或从头创建一个插件：

```
my-plugin/
├── .cursor-plugin/
│   └── plugin.json
├── rules/
│   └── coding-standards.mdc
├── skills/
│   └── code-reviewer/
│       └── SKILL.md
└── mcp.json
```

清单只需要一个 `name` 字段。组件会从其默认目录中自动发现，你也可以在清单中指定自定义路径。

```json
{
  "name": "my-plugin",
  "description": "Custom development tools",
  "version": "1.0.0",
  "author": { "name": "Your Name" }
}
```

**在本地测试插件**

发布前，请从 `~/.cursor/plugins/local` 加载你的插件：

1. 为你的插件创建一个文件夹： `~/.cursor/plugins/local/my-plugin`
2. 将你的插件文件复制到该文件夹中。
3. 重启 Cursor，或运行 Developer: Reload Window。
4. 验证你的插件组件已在 Cursor 中加载，例如规则、技能或 MCP 服务器。

为了更快地迭代，请为你的插件代码仓库创建符号链接：

```shell
ln -s /path/to/my-plugin ~/.cursor/plugins/local/my-plugin
```

## 2.2 Rules

**Rules如何工作**

大语言模型在各次补全之间不会保留记忆。规则在提示层面提供持久、可复用的上下文。

应用规则时，其内容会被加入到模型上下文的开头。这为 AI 在生成代码、理解修改或协助处理工作流程时提供一致的指导。

**规则结构**

每条规则都是一个带有 frontmatter 元数据和内容的 markdown 文件。通过类型下拉菜单控制规则的应用方式，该菜单会更改 `description`、`globs`、`alwaysApply` 属性。

| Rule Type                 | 描述                                      |
| :------------------------ | :---------------------------------------- |
| `Always Apply`            | 应用于每个聊天会话                        |
| `Apply Intelligently`     | 当 Agent 根据描述判断其相关时应用         |
| `Apply to Specific Files` | 当文件匹配指定模式时应用                  |
| `Apply Manually`          | 在聊天中被 @ 提及时应用 (例如 `@my-rule`) |

```markdown
---
globs:
alwaysApply: false
---

- Use our internal RPC pattern when defining services
- Always use snake_case for service names.

@service-template.ts
```
